    Frontend Graph Visualization Improvement Plan¶

    Current Problems¶

    After testing with real queries and exploring the UI, I identified the following UX issues:

    1. Node Status Not Visually Distinct¶

    Problem: Although different styles are defined (seed=amber, expanded=blue, selected=green, evicted=gray), all nodes look very similar in practice. The visual distinction is too subtle.

    Screenshot evidence: In screenshots/step_llm_rerank.png, all nodes appear with similar blue borders, making it impossible to distinguish seeds from expanded nodes.

    2. Edge (Relation) Status Not Visualized¶

    Problem: Currently, only nodes have color-based status. Edges (relations) don't change color based on their status. The LLM-reranked selected relations should be prominently highlighted since they are the most important for answering the query.

    3. Filter/Eviction Step Visualization Missing¶

    Problem: The "Filter Relations" step reduces relations from 63 to 5, but the UI doesn't show which relations were filtered out. Users can't understand what was removed and why.

    4. Timeline-Graph Interaction Not Intuitive¶

    Problem: Clicking different steps does change the graph, but: - No animation/transition showing what changed - Can't easily see what was "added" in each step - The relationship between algorithm steps and visual changes is unclear

    5. Legend Explanation Insufficient¶

    Problem: The bottom legend shows Seed/Expanded/Selected/Filtered, but: - No explanation of what these mean in the algorithm context - Users don't understand the retrieval process from the visualization

    Vector Graph RAG Algorithm Overview¶

    To design better visualization, here's the algorithm flow:

    Query: "When did Lothair II's mother die?"
                    ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 1: Extract Entities                                     │
│   Input: Query text                                          │
│   Output: ["lothair ii"]                                     │
│   Visualization: Show extracted entities as tags             │
└─────────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 2: Retrieve Seeds (Vector Search)                       │
│   - Entity search: Find similar entities → SEED ENTITIES     │
│   - Relation search: Find similar relations → SEED RELATIONS │
│   Output: 2 entities, 20 relations                           │
│   Visualization: Orange/amber nodes and edges                │
└─────────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 3-4: Expand Subgraph (N-degree expansion)              │
│   - From seed entities, follow relations to neighbors        │
│   - From seed relations, find connected entities             │
│   - Repeat for N degrees (typically 1-2)                     │
│   Output: 14 entities, 63 relations                          │
│   Visualization: Blue nodes and edges (expanded)             │
└─────────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 5: Filter Relations (Eviction if too many)             │
│   - If relations > threshold, use vector search to filter    │
│   - Keep most relevant relations                             │
│   Output: 63 → 5 relations (filtered)                        │
│   Visualization: Gray/faded for filtered-out relations       │
└─────────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 6: LLM Rerank                                           │
│   - LLM selects most relevant relations for the query        │
│   Output: 5 selected relations                               │
│   Visualization: Green highlighted edges (CRITICAL PATH)     │
└─────────────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 7: Generate Answer                                      │
│   - Use passages linked to selected relations                │
│   Output: Answer with sources                                │
└─────────────────────────────────────────────────────────────┘

    Important Design Principle: Fixed Graph Structure¶

    Key Insight: The subgraph structure should remain FIXED once loaded. When clicking different timeline steps, only the visual status (colors, opacity) of nodes and edges should change - NOT the graph structure itself.

    This creates a "playback" experience where users can: 1. See the complete expanded subgraph at all times 2. Click any step to see "what was known at that point" 3. Understand how the algorithm progressively discovers and filters information

    Example Behavior:¶

    Final subgraph: 14 entities, 63 relations (structure always visible)

Step 1 "Extract Entities":
  - All nodes/edges: gray/faded (not yet discovered)
  - Query entities highlighted

Step 2 "Retrieve Seeds":
  - Seed entities/relations: orange (newly discovered)
  - Rest: gray/faded

Step 3-4 "Expand":
  - Seed: orange
  - Expanded: blue (newly discovered this step)
  - Rest: gray/faded

Step 5 "Filter":
  - Seed: orange
  - Kept relations: blue
  - Filtered-out relations: gray dashed (still visible but marked as filtered)

Step 6 "LLM Rerank":
  - Selected relations: green highlighted
  - Non-selected: blue or gray
  - Seed: orange

Step 7 "Generate Answer":
  - Show final state with answer path highlighted

    This approach: - Provides spatial consistency (nodes don't jump around) - Shows algorithm progression through color changes - Lets users understand what was filtered vs selected - Reduces cognitive load (one stable layout to learn)
