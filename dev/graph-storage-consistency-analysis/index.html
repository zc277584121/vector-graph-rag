    图存储层一致性问题分析与解决方案¶

    1. 背景¶

    本项目的知识图谱采用的是一种逻辑层图结构，建立在 Milvus 向量数据库之上。图的拓扑关系（点、边、邻接关系）并非由数据库原生约束保证，而是通过三个 Milvus Collection 之间的 ID 互相引用来维护：

    逻辑概念 Milvus Collection 存储的邻接信息
    Entity（实体/点） vgrag_entities relation_ids（关联的边）、passage_ids（来源段落）
    Relation（关系/边） vgrag_relations entity_ids（头尾节点）、passage_ids（来源段落）
    Passage（段落/文本） vgrag_passages entity_ids（包含的实体）、relation_ids（包含的关系）

    核心特点：

    • 图的拓扑完全靠双向 ID 引用维护（例如 Entity 存了 relation_ids，Relation 也存了 entity_ids）
    • Milvus 不支持事务（Transaction）
    • 所有多步骤操作都是非原子的
    • 去重缓存（_entity_name_to_id、_relation_text_to_id）仅存在于内存中，不持久化

    这种设计带来了一系列潜在的一致性问题，本文档对这些问题进行详细分析，并探讨可行的解决方案。

    2. 问题分析¶

    2.1 级联删除的非原子性 — 可能产生悬挂引用（Dangling References）¶

    涉及代码：graph.py 中的 _delete_entity()、_delete_relation()、delete_passage()

    以删除一个 Entity 为例（graph.py:237-282），操作流程如下：

    步骤 1: 从 Milvus 读取 entity 数据 → 得到 relation_ids, passage_ids
步骤 2: 遍历 relation_ids → 逐个更新每个 relation，从其 entity_ids 中移除当前 entity
步骤 3: 遍历 passage_ids → 逐个更新每个 passage，从其 entity_ids 中移除当前 entity
步骤 4: 删除 entity 本身
步骤 5: 更新内存去重缓存 _entity_name_to_id

    风险场景：

    假设该 Entity 关联了 3 个 Relation。步骤 2 成功更新了其中 2 个 Relation 后，程序崩溃或 Milvus 连接中断：

    • 2 个 Relation 已经被修改（对该 Entity 的引用被移除了）
    • 第 3 个 Relation 仍然保留着指向该 Entity 的引用
    • Entity 本身尚未被删除（步骤 4 还没执行）
    • 结果：图处于一个不一致的中间状态。已修改的 2 个 Relation 无法回滚，操作既没完成也没撤销

    同样的问题存在于 _delete_relation()（graph.py:430-475）和 delete_passage()（graph.py:634-672）中。

    严重程度：高 发生概率：中（取决于网络稳定性和 Milvus 可用性）

    2.2 创建操作的非原子性 — 可能产生孤儿节点或悬挂引用¶

    涉及代码：graph.py 中的 create_passage()、_create_relation()

    以 create_passage() 为例（graph.py:479-554），操作流程如下：

    步骤 1: 生成 passage embedding
步骤 2: 遍历 triplets:
    → _create_relation(subject, predicate, object, passage_ids=[passage_id])
        → _create_entity(subject)    ← 插入 entity 到 Milvus
        → _create_entity(object)     ← 插入 entity 到 Milvus
        → 插入 relation 到 Milvus（此时 relation 已经记录了 passage_id）
步骤 3: 插入 passage 到 Milvus

    风险场景 A：步骤 2 成功创建了 Entity 和 Relation，但步骤 3 插入 Passage 失败（Milvus 磁盘满、网络超时等）：

    • Entity 和 Relation 已经存在于 Milvus 中
    • 这些 Entity 和 Relation 的 passage_ids 字段引用了一个不存在的 passage_id
    • 结果：图中存在悬挂引用，子图扩展时 _fetch_passages() 查不到该 Passage

    风险场景 B：在 _create_relation() 内部（graph.py:328-350），先创建了两个 Entity（步骤成功），然后插入 Relation 失败：

    • 两个 Entity 的 relation_ids 字段指向一个不存在的 Relation ID
    • 结果：图遍历时会尝试获取不存在的 Relation，返回不完整的子图

    严重程度：高 发生概率：中

    2.3 内存去重缓存与 Milvus 状态不一致¶

    涉及代码：graph.py:93-94

    self._entity_name_to_id: Dict[str, str] = {}   # 内存缓存
self._relation_text_to_id: Dict[str, str] = {}  # 内存缓存

    这两个去重映射是纯内存的，不会持久化到任何存储中。这带来三个子问题：

    问题 A — 进程重启后缓存丢失导致数据重复：

    当 Graph 对象被重新创建（进程重启、新建实例等），缓存丢失，但 Milvus 中的数据还在。此时再插入相同名字的 Entity，由于缓存中找不到该名字，会导致同名重复 Entity 被创建到 Milvus 中。

    问题 B — Milvus 写入成功但缓存未更新：

    在 _create_entity() 中（graph.py:153-162）：

    # 先写 Milvus
self._store._insert_entities(...)
# 然后才更新缓存
self._entity_name_to_id[normalized_name] = entity_id

    如果 Milvus 写入成功但在更新缓存前程序崩溃，重启后缓存没有此条目，会再次向 Milvus 插入重复数据。

    问题 C — 多实例缓存不同步：

    如果有多个 Graph 实例指向同一个 Milvus，各自维护独立的内存缓存，无法感知对方的写入操作。

    严重程度：中 发生概率：高（只要进程重启就必然发生）

    2.4 add_documents() 全量替换的先删后建风险¶

    涉及代码：rag.py:354-383

    # 先删除所有 collection（旧数据立即消失）
self._store.drop_collections()
self._store.create_collections(drop_existing=True)

# 然后依次插入新数据
self._store._insert_entities(...)     # 步骤 1
self._store._insert_relations(...)    # 步骤 2
self._store.insert_passages(...)      # 步骤 3

    风险场景：Collection 已经被 drop（旧数据全部丢失），步骤 1 成功插入了 Entity，但步骤 2 插入 Relation 时失败：

    • 旧数据已经没了，无法恢复
    • 新数据只写入了一部分（只有 Entity，没有 Relation 和 Passage）
    • 结果：整个知识图谱处于不可用状态，且完全无法回滚

    严重程度：高 发生概率：低（全量导入通常在可控环境下进行，但一旦发生后果严重）

    2.5 并发访问的竞态条件（Race Condition）¶

    涉及代码：graph.py 中所有的读-改-写操作

    以 _create_entity() 中的去重更新为例（graph.py:124-138）：

    # 读取当前状态
existing = self._store._get_entities_by_ids([existing_id])
current = existing[0]
# 在本地合并
new_relation_ids = list(set(current.get("relation_ids", []) + (relation_ids or [])))
# 写回
self._store._update_entity(existing_id, relation_ids=new_relation_ids)

    风险场景：两个线程/进程同时为同一个 Entity 添加不同的 Relation：

    线程 A: 读取 entity.relation_ids = [R1, R2]
线程 B: 读取 entity.relation_ids = [R1, R2]     ← 读到相同状态
线程 A: 合并后写入 [R1, R2, R3]
线程 B: 合并后写入 [R1, R2, R4]                  ← 覆盖了线程 A 的结果
最终结果: [R1, R2, R4]（R3 丢失了）

    严重程度：中 发生概率：低（目前主要是单线程使用，但如果引入 Web 服务或并行处理就会出现）

    2.6 子图扩展时的脏读（Stale Read）¶

    涉及代码：knowledge_graph.py:258-356 中的 SubGraph.expand()

    子图扩展是一个多步骤的懒加载过程，跨越多次 Milvus 查询：

    初始化: 获取种子 entity → 从 Milvus 读取 relation_ids
第 1 跳: 通过 relation_ids 从 Milvus 获取 relation → 读取 entity_ids → 从 Milvus 获取新 entity
第 2 跳: 通过新 entity 从 Milvus 获取更多 relation → ...
最后: 收集所有 passage_ids → 从 Milvus 获取 passage

    风险场景：在扩展过程中，另一个操作正在修改图（删除一个 Entity、更新一个 Relation 等），那么扩展过程中不同步骤读到的图可能处于不一致的状态：

    • 步骤 1 读到的 Entity 引用了某些 Relation
    • 到步骤 2 去获取这些 Relation 时，它们可能已经被删除了

    代码中有 if relation: 之类的空值检查，不会崩溃，但返回的子图可能是不完整或自相矛盾的。

    严重程度：低 发生概率：低（需要读写并发才会触发）

    2.7 双向引用的维护负担¶

    图的邻接关系采用了双向冗余存储：

    • Entity 存了 relation_ids ↔ Relation 存了 entity_ids
    • Entity 存了 passage_ids ↔ Passage 存了 entity_ids
    • Relation 存了 passage_ids ↔ Passage 存了 relation_ids

    每次修改操作都需要同时更新双方。由于没有事务保证，任何一次更新失败都会导致双向引用不对称。例如：Entity A 说它连着 Relation R，但 Relation R 的 entity_ids 里已经没有 A 了。

    这种不对称不会导致程序崩溃，但会导致从不同方向遍历图得到不同的结果。

    严重程度：中 发生概率：中（与问题 2.1、2.2 联动发生）

    3. 问题影响总结¶

    问题 严重程度 发生概率 主要影响
    2.1 级联删除非原子 高 中 图处于不一致的中间状态
    2.2 创建操作非原子 高 中 悬挂引用、孤儿数据
    2.3 内存缓存不持久 中 高 重启后数据重复
    2.4 全量替换先删后建 高 低 数据全丢且不可恢复
    2.5 并发竞态 中 低 引用丢失
    2.6 子图扩展脏读 低 低 查询结果不完整
    2.7 双向引用不对称 中 中 不同方向遍历结果不一致

    在当前 "一次全量构建 + 多次只读查询" 的主要使用模式下，问题 2.4（全量替换）和问题 2.3（缓存不持久）是最需要关注的。如果未来引入增量更新或多用户并发场景，问题 2.1、2.2、2.5 会变得非常突出。

    4. 解决方案探讨¶

    4.1 方案 A：正统的事务方案 — 对本项目不现实¶

    方案 难度 评估
    Milvus 原生事务 — Milvus 不支持事务，无从谈起
    Saga 补偿事务模式 极高 每个操作需要编写对应的 undo 操作，引入状态机，代码量翻倍，杀鸡用牛刀
    WAL（Write-Ahead Log） 高 写操作前先记录日志，失败后重放或回滚。需要额外的持久化层，复杂度过高
    更换为支持事务的图数据库 高 如 Neo4j、TigerGraph 等。但违背了项目"纯 Milvus"的设计初衷，且引入了重量级外部依赖

    结论：这些方案对于本项目的规模和定位来说投入产出比不合理。

    4.2 方案 B：低成本的针对性修复¶

    以下几个改动成本低、收益明确，可以直接实施：

    B1. add_documents() 改为先建后删¶

    当前做法（危险）：

    self._store.drop_collections()           # 旧数据立即消失
self._store.create_collections()
self._store._insert_entities(...)        # 如果这里失败，旧数据已经没了
self._store._insert_relations(...)
self._store.insert_passages(...)

    改进方案：

    # 1. 先用临时名称创建新 collection
tmp_prefix = f"_tmp_{timestamp}_"
# 向临时 collection 插入所有数据
insert_entities_to(tmp_prefix + "entities", ...)
insert_relations_to(tmp_prefix + "relations", ...)
insert_passages_to(tmp_prefix + "passages", ...)

# 2. 全部成功后，删除旧 collection，重命名新 collection
drop_collections(old_prefix)
rename_collections(tmp_prefix, target_prefix)

    或者更简单的方案：

    # 1. 创建新 collection（使用新名称或带版本号的名称）
# 2. 插入所有数据
# 3. 全部成功后，原子性地切换引用（更新 store 中的 collection 名称）
# 4. 删除旧 collection

    成本：低 收益：避免最严重的"数据全丢"风险

    B2. 启动时从 Milvus 重建内存去重缓存¶

    def _warm_up_cache(self):
    """从 Milvus 读取已有 entity/relation，重建去重缓存。"""
    # 重建 entity 缓存
    all_entities = self._store._get_all_entities()  # 需要新增此方法
    for e in all_entities:
        name = e.get("text", "")
        self._entity_name_to_id[name] = e["id"]

    # 重建 relation 缓存
    all_relations = self._store._get_all_relations()
    for r in all_relations:
        text = r.get("text", "")
        self._relation_text_to_id[text] = r["id"]

    在 Graph.__init__() 中可选地调用此方法。对于大规模图可以加一个开关，默认开启。

    成本：低 收益：解决进程重启后重复插入的问题

    B3. 删除操作增加 try-catch 和最大努力日志¶

    def _delete_entity(self, entity_id: str) -> bool:
    entities = self._store._get_entities_by_ids([entity_id])
    if not entities:
        return False

    entity_data = entities[0]
    failed_updates = []

    # 级联更新 relation
    for rid in entity_data.get("relation_ids", []):
        try:
            relations = self._store._get_relations_by_ids([rid])
            if relations:
                new_ids = [eid for eid in relations[0].get("entity_ids", []) if eid != entity_id]
                self._store._update_relation(rid, entity_ids=new_ids)
        except Exception as e:
            failed_updates.append(("relation", rid, str(e)))

    # 级联更新 passage（同理）
    # ...

    # 删除 entity 本身
    result = self._store._delete_entity(entity_id)

    if failed_updates:
        logger.warning(f"Entity {entity_id} 级联删除部分失败: {failed_updates}")

    return result

    成本：低 收益：不能保证一致性，但至少能发现问题、方便排查

    4.3 方案 C：增加一致性检查与修复工具¶

    不在每次操作时预防不一致，而是提供事后检测和修复的能力：

    class ConsistencyChecker:
    def check(self) -> ConsistencyReport:
        """扫描整个图，检测所有不一致问题。"""
        issues = []

        # 检查 1: Entity.relation_ids 引用的 Relation 是否都存在
        for entity in all_entities:
            for rid in entity.relation_ids:
                if not relation_exists(rid):
                    issues.append(DanglingReference("entity", entity.id, "relation", rid))

        # 检查 2: Relation.entity_ids 引用的 Entity 是否都存在
        for relation in all_relations:
            for eid in relation.entity_ids:
                if not entity_exists(eid):
                    issues.append(DanglingReference("relation", relation.id, "entity", eid))

        # 检查 3: 双向引用对称性
        # Entity A 引用了 Relation R → Relation R 也应该引用 Entity A
        for entity in all_entities:
            for rid in entity.relation_ids:
                relation = get_relation(rid)
                if relation and entity.id not in relation.entity_ids:
                    issues.append(AsymmetricReference("entity", entity.id, "relation", rid))

        # 检查 4: Passage 引用的 Entity/Relation 是否存在
        # ...

        return ConsistencyReport(issues=issues)

    def repair(self, report: ConsistencyReport):
        """根据检查报告修复不一致。"""
        for issue in report.issues:
            if isinstance(issue, DanglingReference):
                # 移除悬挂引用
                remove_reference(issue.source_type, issue.source_id, issue.target_type, issue.target_id)
            elif isinstance(issue, AsymmetricReference):
                # 补充缺失的反向引用
                add_back_reference(issue.source_type, issue.source_id, issue.target_type, issue.target_id)

    成本：中 收益：不影响正常操作的性能和复杂度，用户可按需执行，作为兜底机制

    4.4 方案 D：文档说明 + 使用约束¶

    对于当前阶段来说最务实的做法：

    1. 在文档中明确说明：这是逻辑层的图结构，不支持事务，增量操作存在一致性风险
    2. 推荐使用模式：优先使用 add_documents() 全量构建，避免频繁的增量 CRUD 操作
    3. 单实例使用：不要多个进程同时操作同一个 Milvus 实例的同一组 Collection

    5. 引入外部图存储方案的可行性分析¶

    除了在现有架构上修补，我们还调研了将图拓扑存储迁移到专门的图存储引擎的可行性。

    5.1 调研范围¶

    调研覆盖了以下类型的图存储方案：

    • 纯内存图算法库：NetworkX、igraph、rustworkx、graph-tool
    • 嵌入式数据库方案：SQLite（自定义 schema）、DuckDB + DuckPGQ
    • 独立图数据库：Neo4j、Memgraph、ArangoDB、TigerGraph
    • 新兴嵌入式图数据库：KuzuDB、FalkorDBLite、CogDB

    5.2 纯内存图库：无法解决核心问题¶

    库 事务支持 持久化 安装方式 结论
    NetworkX 无 无（手动 pickle/JSON） pip install networkx 只是把非原子问题从 Milvus 搬到内存里，本质没变
    igraph 无 无（手动序列化） pip install igraph 同上，图算法性能更好但不解决一致性问题
    rustworkx 无 无 pip install rustworkx 同上，Rust 实现性能最优
    graph-tool 无 无 需要编译 C++（安装复杂） 同上，且安装门槛过高

    结论：这些是图算法库，不是图存储方案。它们没有事务、没有持久化机制、没有并发控制。如果用它们来存储图拓扑，只是把 Milvus 上的一致性问题换了一个地方重现，核心矛盾没有任何缓解。

    5.3 SQLite（自定义 Schema）— 最推荐方案¶

    核心优势：

    特性 评估
    ACID 事务 完整支持。所有级联操作可以包在 BEGIN/COMMIT 中，要么全部成功要么全部回滚
    部署成本 零。Python 标准库自带 sqlite3，无需任何额外安装
    运维成本 极低。单个 .db 文件，与现有 Milvus .db 并列存放
    外键级联删除 原生支持 ON DELETE CASCADE，数据库自动处理引用清理
    Python 集成 sqlite3 是 Python 最成熟的数据库模块
    性能 对 10k-500k 节点规模的知识图谱完全足够

    改造后的架构分工：

    ┌─────────────────────┐    ┌─────────────────────────┐
│       Milvus        │    │         SQLite           │
│                     │    │                          │
│  - 向量 Embedding   │    │  - Entity 表             │
│  - ANN 相似度搜索   │    │  - Relation 表           │

│                     │    │  （做它擅长的事）         │

    示意 Schema：

    -- 开启外键支持
PRAGMA foreign_keys = ON;

CREATE TABLE entities (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL UNIQUE
);

CREATE TABLE relations (
    id TEXT PRIMARY KEY,
    subject_id TEXT NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
    object_id TEXT NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
    predicate TEXT NOT NULL,
    text TEXT NOT NULL
);

CREATE TABLE passages (
    id TEXT PRIMARY KEY
);

-- 多对多关联表：Entity ↔ Passage
CREATE TABLE entity_passages (
    entity_id TEXT NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
    passage_id TEXT NOT NULL REFERENCES passages(id) ON DELETE CASCADE,
    PRIMARY KEY (entity_id, passage_id)
);

-- 多对多关联表：Relation ↔ Passage
CREATE TABLE relation_passages (
    relation_id TEXT NOT NULL REFERENCES relations(id) ON DELETE CASCADE,
    passage_id TEXT NOT NULL REFERENCES passages(id) ON DELETE CASCADE,
    PRIMARY KEY (relation_id, passage_id)
);

    改造前后对比 — 删除 Entity 操作：

    改造前（graph.py:237-282，30+ 行，非原子）：

    def _delete_entity(self, entity_id: str) -> bool:
    entities = self._store._get_entities_by_ids([entity_id])
    if not entities:
        return False
    entity_data = entities[0]
    # 手动逐个更新每个关联的 relation（可能部分失败）
    for rid in entity_data.get("relation_ids", []):
        relations = self._store._get_relations_by_ids([rid])
        if relations:
            new_entity_ids = [eid for eid in relations[0].get("entity_ids", []) if eid != entity_id]
            self._store._update_relation(rid, entity_ids=new_entity_ids)
    # 手动逐个更新每个关联的 passage（可能部分失败）
    for pid in entity_data.get("passage_ids", []):
        passages = self._store.get_passages_by_ids([pid])
        if passages:
            new_entity_ids = [eid for eid in passages[0].get("entity_ids", []) if eid != entity_id]
            self._store.update_passage(pid, entity_ids=new_entity_ids)
    # 最后才删除 entity
    result = self._store._delete_entity(entity_id)
    return result

    改造后（完全原子）：

    def _delete_entity(self, entity_id: str) -> bool:
    with self.sqlite_conn:  # 自动事务：成功 commit，异常 rollback
        cursor = self.sqlite_conn.execute(
            "DELETE FROM entities WHERE id = ?", (entity_id,)
        )
        return cursor.rowcount > 0
    # ON DELETE CASCADE 自动清理：
    #   - relations 表中 subject_id 或 object_id 为该 entity 的行
    #   - entity_passages 表中 entity_id 为该 entity 的行

    能解决的问题：

    原问题 SQLite 方案的解决效果
    2.1 级联删除非原子 完全解决 — ON DELETE CASCADE + 事务
    2.2 创建操作非原子 完全解决 — 所有写操作包在一个事务里
    2.3 内存缓存不持久 完全解决 — 去重直接通过 UNIQUE 约束 + INSERT OR IGNORE 实现
    2.4 全量替换先删后建 完全解决 — 事务保证要么全部替换成功，要么回滚
    2.5 并发竞态 基本解决 — SQLite WAL 模式支持并发读、串行写
    2.6 子图扩展脏读 缓解 — 单个事务内读取保证一致性快照
    2.7 双向引用不对称 完全解决 — 关系通过外键和关联表维护，无需双向冗余存储

    5.4 DuckDB + DuckPGQ — 次选方案¶

    特性 评估
    ACID 事务 完整支持
    部署成本 低（pip install duckdb）
    图查询语法 DuckPGQ 扩展提供 SQL/PGQ 标准图查询语法
    性能 面向分析型负载优化，图查询性能好

    优势：比 SQLite 多了图查询语法（SQL/PGQ 标准），如果后续需要复杂的图分析查询（社区发现、路径计算等）会更方便。

    劣势：DuckPGQ 是社区扩展而非核心功能，成熟度和稳定性不如 SQLite；API 比 SQLite 复杂。

    5.5 Neo4j — 功能最强但太重¶

    特性 评估
    ACID 事务 完整支持，Cypher 查询语言
    向量搜索 2026 年已原生支持（理论上可同时替代 Milvus + 图层）
    部署成本 高。必须运行独立的 JVM 服务进程，没有 Python 嵌入模式

    结论：功能上完全可以解决所有问题，甚至可以替代 Milvus。但对于一个 Python 库项目来说，要求用户额外部署一个 JVM 服务是不可接受的运维负担。

    5.6 其他方案¶

    方案 状态 评估
    KuzuDB 2025 年 10 月已 archive 曾经是最理想的嵌入式图数据库（Cypher + 向量搜索），但已废弃，不建议使用
    FalkorDBLite 新项目（2025 年） 嵌入式图数据库，支持 Cypher + 向量搜索。但需要 Python 3.12+，底层 fork Redis 子进程，成熟度不足。值得 6-12 个月后再关注
    CogDB 小众项目 纯 Python 嵌入式图数据库，支持向量搜索。但无事务支持，社区很小，不适合生产使用
    Memgraph 成熟 内存型图数据库，完整 ACID。但和 Neo4j 一样需要独立服务进程，部署太重
    ArangoDB 成熟 多模型数据库（文档 + 图 + KV），完整 ACID。同样需要独立服务进程
    TigerGraph 企业级 部署极重，不适合库级项目

    5.7 综合对比¶

    方案 解决事务问题 部署复杂度 Python 支持 需保留 Milvus 推荐度
    SQLite 完全解决 零（标准库） 最好 是 强烈推荐
    DuckDB+DuckPGQ 完全解决 很低（pip） 好 是 次选
    Neo4j 完全解决 高（独立服务） 好 可选 不推荐（太重）
    NetworkX 不解决 零（pip） 最好 是 不推荐
    FalkorDBLite 基本解决 中（子进程） 一般 可选 暂不推荐（不成熟）

    6. 建议的实施路径¶

    综合成本、收益和紧迫性，建议按以下优先级逐步实施：

    第一阶段（P0）：文档说明 + 低成本修复¶

    1. 文档说明（方案 D）：在项目文档中明确说明逻辑图层的限制和推荐使用模式
    2. add_documents() 先建后删（方案 B1）：避免最严重的数据丢失风险
    3. 启动时重建去重缓存（方案 B2）：避免进程重启后的数据重复问题

    第二阶段（P1）：引入 SQLite 图存储¶

    1. 设计 SQLite Schema（参考 5.3 节）
    2. 实现 SQLite 存储层，替换当前 Milvus 中的图拓扑存储
    3. Milvus 仅保留向量 Embedding 和 ANN 搜索功能
    4. 利用 SQLite 事务和外键约束，简化所有 CRUD 操作

    第三阶段（P2）：增强工具¶

    1. 实现一致性检查与修复工具（方案 C）
    2. 删除操作增加日志记录（方案 B3）
